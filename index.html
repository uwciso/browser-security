Open up the app.js file in a text editor, and for each CSP challenge below, create a corresponding content security policy by editing the csp_directives object. Refer to the Helmet.js docs for format/syntax guidance.

X-XSS-Protection HeaderMDN

  1. As an added layer of protection for older browsers not supporting CSP, make sure the X-XSS-Protection header is set for all responses. This can be done by adding a single line to the app.js file using Helmet.js's XSS Filter module.

    Reload the page and verify this header is indeed set.

Subresource Integrity MDN

  1. Right now, this page pulls in jQuery from a CDN, and in doing so, is trusting that the CDN will serve an uncorrupted file. If the file was modified, the browser would have no way to detect that. Add another layer of security for this file by adding an integrity attribute on the script tag of this file, index.html. For the attribute's value, you can easily generate a hash of the (jQuery) file using this SRI Hash Generator. Don't forget to include a crossorigin="anonymous" attribute as well (see MDN for more information on this attribute).

    Now, reload the page and verify that the resource loads as expected. Then, modify the value of the integrity attribute so the browser's calculated value will not match. Has the resource been blocked?

    Note: verifying this can be tricky, since the Network tab will show the file as having been loaded (which is true, since the file had to be downloaded in order for the browser to check its hash). You can verify by attempting to execute a jQuery selection in the console window, or, in Chrome Dev Tools, using the Source tab to see if that file is available.

  2. Now, with the integrity attribute correctly applied, remove the crossorigin="anonymous" attribute completely, and reload the page. Does the script resource load? Why or why not?

Explicit MIME Types and Disabling MIME SniffingMDN

  1. For each of the local page resources, check the response to verify a ContentType header was sent (use the Dev Tools Network tab). Also, verify whether or not browser MIME sniffing has been explicitly disabled by the server (to prevent the browser potentially making its own guess at the resource's content type and ignoring information sent by the server). If it hasn't, fix this using Helmet.js's noSniff module (you'll need to add a line in the app.js file).

    Reload the page, and verify the presence of the appropriate response header (X-Content-Type-Options: nosniff).

Safer CookiesMDN

For these exercises, you'll need to edit the res.cookie method in the app.js file.

  1. Note that this app sets a 'chipsahoy' cookie on the initial response. Use document.cookie in the Dev Tools console to see this cookie and its value. Now, add the httpOnly directive to this cookie to prevent it from being accessed directly with JS methods.

    Reload the page and try accessing the cookie again through the console with document.cookie.

  2. Check that this cookie is sent on subsequent requests to local (same-origin) resources (such as this page's stylesheet or image) by inspecting the Cookie request header. Now, disallow this cookie from being sent on requests not over https by using the secure directive.

    Reload the page, then check those same requests to verify this cookie was not sent (since this app is making the requests over http rather than https).

  3. Finally, ensure that this cookie won't be sent along with any cross-site (third party) requests by adding the sameSite directive. Also, give this cookie an explicit expiration date (HINT: an easy way to do that is to use new Date(Date.now + 1000) as a value).

    Reload the page and check that the cookie reflects these updates.
